Introduction & Project Overview

The Smart City Management System (SCMS) is a full-stack, mobile-first digital platform designed to empower urban citizens to report, track, and engage with city infrastructure issues — while giving government and city authorities a powerful web-based dashboard to manage and resolve those complaints effectively.

Urban areas across the world face ongoing challenges with infrastructure degradation. Roads develop potholes, street lights fail, drains overflow, and illegal dumping occurs at alarming frequencies. Traditional systems — phone lines, paper forms, static email portals — are slow, unacknowledged, and opaque. SCMS was built to replace these outdated workflows with a transparent, real-time, GPS-verified, photo-backed digital complaint ecosystem.

💡

Core Mission: SCMS connects every citizen directly to city authorities through a modern Android application, ensuring that every infrastructure problem is reported, tracked, prioritized, and resolved — with full transparency for both parties.

What Makes SCMS Different?

  • Works without internet — complaints are saved locally and synced automatically when connectivity returns
  • GPS + Photo verified — every complaint carries geolocation data and visual evidence, making fake or vague reports impossible
  • AI-lite duplicate detection — the backend detects if a similar complaint already exists nearby and gives citizens the option to upvote instead of duplicating
  • Peer-to-Peer mesh relay — in extreme offline scenarios, complaints are relayed device-to-device via Bluetooth/Wi-Fi until they reach a connected peer
  • Gamified participation — citizens earn XP points and badges for reporting, creating a healthy competitive civic culture
  • Real-time alerts — city officials can broadcast emergency alerts instantly to all citizens via push notifications

All Technologies Used — What & Why

🗄️ Data Storage Technologies

🗃️

MySQL

Relational DB

Used for: Storing all structured complaint data — complaint ID, category, address, latitude, longitude, description, user ID, status, photo URLs, upvotes, timestamps.

Why MySQL? Complaints have a fixed schema with relationships (user → complaints, complaint → photos). MySQL's relational model handles this perfectly with SQL queries, and it integrates seamlessly with the Node.js backend.

🔥

Firebase Cloud Firestore

NoSQL Cloud DB

Used for: Storing user profile data — name, email, total points (XP), badge level, registration date.

Why Firestore? User profile data is unstructured and updated frequently (points increment with each complaint). Firestore's real-time document model handles this better than MySQL for user-centric data.

📱

Android SharedPreferences

Local Storage

Used for: (1) Persisting the logged-in user's session data so the app doesn't ask for login every time. (2) Storing the offline complaint queue (JSON) when no internet is available.

Why SharedPreferences? Lightweight, fast key-value store built into Android. Perfect for session tokens and small JSON payloads that need to survive app closes.

☁️

Cloudinary CDN

Media Storage

Used for: Storing and serving all complaint photos. Photos are uploaded from the Android app → Node.js backend → Cloudinary, and the resulting URL is stored in MySQL.

Why Cloudinary? Cloudinary provides a globally distributed CDN, automatic image compression, resizing, and format optimization. It offloads the server from having to serve heavy image files directly.

🔐 Authentication Technology

🔑

Firebase Authentication

Auth Service

Used for: User registration, email/password login, and session management. On first login, Firebase returns a UID and auth token tied to the user's account permanently.

Why Firebase Auth? Industry-standard, battle-tested authentication service. Handles password hashing, token refresh, account recovery, and session validity — eliminating the need to build custom auth from scratch.

🧠

SessionManager (Custom)

Session Layer

Used for: Persisting the user's profile (name, email, ID, points) to SharedPreferences so the app can restore the session on next launch without needing a network call.

Why custom? Firebase Auth alone only stores the auth token, not profile data. SessionManager bridges this gap by locally caching the full Firestore user profile.

🌐 Networking Technologies

📡

Retrofit 2 + OkHttp 4

HTTP Client

Used for: All API calls from Android — submitting complaints (multipart), fetching complaint lists, upvoting, checking duplicates, fetching alerts, and leaderboard data.

Why Retrofit? Type-safe HTTP client for Android/Kotlin. Automatically serializes/deserializes JSON, handles multipart file uploads, and works perfectly with Kotlin coroutines for non-blocking async calls.

⚡

Node.js + Express.js

REST API Server

Used for: The complete backend REST API — complaint CRUD, duplicate detection, photo upload to Cloudinary, alert management, leaderboard calculation, and user management.

Why Node.js? Non-blocking, event-driven architecture handles multiple concurrent API requests efficiently. Express.js makes API route definition clean and concise.

📱 Android UI & Framework

🎨

Jetpack Compose + Material 3

UI Framework

Used for: Building all 12+ screens of the Android app — Login, Signup, Dashboard, Report, Complaints, Alerts, Profile, Leaderboard, Admin, SOS, Camera, and Complaint Timeline.

Why Compose? Google's modern declarative UI toolkit for Android. No XML layouts — UI is built with composable functions in Kotlin. Supports animations, Material 3 design system, and reactive state management out of the box.

🔀

Jetpack Navigation Compose

Navigation

Used for: Managing all screen transitions — login → home → report → camera → back stack management. Also handles passing arguments between screens (e.g., complaint ID to complaint detail).

Why Jetpack Nav? Official Google navigation solution for Compose. Handles the back stack, deep links, and argument passing with type safety.

📸 Camera & Location

📸

CameraX

Camera API

Used for: Providing a live camera viewfinder inside the app for capturing complaint photos. Supports flash toggle, lifecycle-aware binding, and JPEG output.

Why CameraX? Modern replacement for the deprecated Camera2 API. Lifecycle-aware, consistent across Android versions, and optimized for Compose integration.

📍

FusedLocationProviderClient

GPS

Used for: Getting real-time GPS coordinates (latitude/longitude) for every complaint submission. Geocoder converts raw coordinates → human-readable street address.

Why Fused Location? Google's unified location API automatically chooses the best available location provider (GPS, network, sensors) for maximum accuracy and battery efficiency.

🗺️

OSMDroid (OpenStreetMap)

Map Display

Used for: Displaying an interactive map view showing the geotagged location of complaints.

Why OSMDroid? Free, open-source map library — no API key costs unlike Google Maps. Works offline with tile caching.

🔔 Notifications & Mesh Networking

🔔

Firebase Cloud Messaging (FCM)

Push Notifications

Used for: Broadcasting emergency alerts and civic announcements from the Admin Dashboard to all SCMS users simultaneously, even when the app is backgrounded.

Why FCM? Google's free, reliable push notification service. Supports topic-based broadcasting (all users = all_users topic), guaranteed delivery, and Android background service integration.

📶

Google Nearby Connections API

P2P Mesh

Used for: Creating a Peer-to-Peer mesh network between nearby SCMS devices via Bluetooth and Wi-Fi Direct. Offline complaints can be relayed device-to-device until they reach a node with internet access.

Why Nearby Connections? Abstracts Bluetooth, Wi-Fi Direct, and NFC into a single clean API. Uses P2P_CLUSTER strategy for a true mesh topology with automatic connection and discovery management.

🖥️ Admin Dashboard & Backend Utilities

⚛️

React.js (Vite)

Admin Web App

Used for: The city authority admin dashboard — view all complaints, filter by status/category, change complaint status, send alerts to all users, view photos and geotagged locations.

Why React? Component-based architecture makes the dashboard modular and maintainable. Vite provides fast development HMR. Works with the same Node.js REST API the Android app uses.

📦

Nodemailer + bcrypt + JWT

Backend Utilities

Nodemailer: Used for sending SMTP email notifications (e.g., account verification, password resets).

bcrypt: Password hashing for secure admin credential storage.

JWT: Stateless auth tokens for admin dashboard API access.

Rana, R. K., Chou, C. T., Kanhere, S. S., Bulusu, N., & Hu, W. (2010). IPSN '10.

Introduced the concept of participatory sensing — using mobile users as distributed sensor nodes for urban data collection. Demonstrated that commodity smartphones are viable urban monitoring tools via embedded GPS and cameras.

🔗 SCMS Relevance: SCMS directly implements participatory sensing — every citizen with the app becomes a node in the urban monitoring network, reporting geo-tagged, photo-verified complaints.

Internet of Things for Smart Cities

Zanella, A., Bui, N., Castellani, A., Vangelista, L., & Zorzi, M. (2014). IEEE IoT Journal, 1(1).

Outlined how IoT platforms can monitor and manage urban infrastructure by connecting physical assets to digital management systems. Emphasized real-time data, centralized dashboards, and citizen-authority feedback loops.

🔗 SCMS Relevance: SCMS implements this closed-loop model — citizen reports flow to the MySQL database, visible in the React admin dashboard, with status updates feeding back to citizens.
Paper 03 · Offline-First Architecture

Cloud Computing for Enhanced Mobile Health Applications

Nkosi, M. T., & Mekuria, F. (2010). IEEE CloudCom 2010.

Proposed offline-first mobile architectures — data captured locally and synchronized to the cloud when connectivity is available. Addressed local persistence and conflict resolution strategies for intermittent connectivity environments.

🔗 SCMS Relevance: OfflineComplaintManager implements exactly this — JSON complaint serialization to SharedPreferences, with automatic sync via ConnectivityManager.NetworkCallback when internet returns.
Paper 04 · Duplicate Detection

AIDR: Artificial Intelligence for Disaster Response

Imran, M., Castillo, C., Lucas, J., Meier, P., & Vieweg, S. (2014). WWW '14.

Analyzed redundant report clustering in crowdsourced crisis systems. Proposed geographic and categorical deduplication to reduce administrative overhead while enabling community-validated incident priority.

🔗 SCMS Relevance: Backend /complaint/check-duplicate API checks incoming GPS+category against existing records and returns isDuplicate, triggering the upvote dialog in-app.
Paper 05 · Push Notification Systems

Opportunistic Privacy in Mobile Social Applications

Kapadia, A., Myers, S., Wang, X., & Wetzel, R. (2009). UIST '09.

Explored push notification systems as real-time civic communication channels connecting event publishers (authorities) to targeted citizen subscriber groups, particularly for time-sensitive alerts.

🔗 SCMS Relevance: FCM topic-based messaging (all_users) allows admins to instantly push emergency alerts. MyFirebaseMessagingService handles delivery even when the app is in background.
Paper 06 · Peer-to-Peer Mesh Networking

On the Levy-Walk Nature of Human Mobility

Rhee, I., Shin, M., Hong, S., Lee, K., Kim, S. J., & Chong, S. (2011). IEEE/ACM Transactions on Networking, 19(3).

Demonstrated that human mobility patterns provide a natural substrate for opportunistic P2P data exchange. In infrastructure-damaged or internet-denied areas, devices following human movement patterns can successfully relay data node-to-node.

🔗 SCMS Relevance: MeshNetworkManager uses Google Nearby Connections (P2P_CLUSTER, Bluetooth+WiFi Direct) to relay offline complaints between SCMS users until one with internet receives and uploads them.

Proposed Methodology

Development Approach

SCMS was developed using an Agile-inspired iterative approach — beginning with core authentication and complaint submission, then progressively adding offline mode, gamification, mesh networking, and the admin dashboard in distinct development sprints.

Module Implementation Details

Module 1: Authentication & Persistence

Firebase Auth manages email/password login. On successful authentication, the app fetches the user's extended profile from Cloud Firestore and writes it to SharedPreferences via SessionManager. On every app launch, MainActivity checks FirebaseAuth.currentUser AND SessionManager.getUser() — if both exist, the user is silently auto-logged in and taken directly to the Dashboard. UserSession is a Kotlin singleton holding the in-memory current user object accessible across all screens.

Module 2: Complaint Reporting Pipeline

The complaint flow: (1) User opens camera via CameraX, captures photos, (2) FusedLocationClient retrieves GPS coordinates in the background, (3) Geocoder converts to address, (4) User selects category and writes description ≥10 characters, (5) On submit, RetrofitClient.api.checkDuplicateComplaint() is called, (6) If no duplicate, complaint is sent as MultipartBody with all photos and form fields to the Node.js backend, (7) Backend uploads photos to Cloudinary and stores the complaint in MySQL.

Module 3: Offline System

NetworkUtils.isInternetAvailable() uses ConnectivityManager and NetworkCapabilities to detect live connectivity. When offline and the user submits a complaint, OfflineComplaintManager.saveComplaintOffline() serializes the complaint to JSON using Gson and appends it to the SharedPreferences complaint queue. In MainActivity, a ConnectivityManager.NetworkCallback fires on network availability, triggering OfflineComplaintManager.syncOfflineComplaints() to upload all queued complaints to the server.

Module 4: Gamification

Each verified complaint submission increments the user's points field in their Firestore document. Badge levels are computed as: Bronze (0–50 pts), Silver (51–150 pts), Gold (151+ pts). The Leaderboard screen fetches all user records from the backend, sorted by total points, and renders them in a ranked list.

Module 5: Push Notifications

FCM is initialized in MainActivity which subscribes the device to the all_users FCM topic. When an admin creates an alert via the web dashboard, the Node.js backend calls the FCM API with the topic message. MyFirebaseMessagingService handles message reception in background and foreground, displaying a system notification with the alert title and body.

Module 6: Mesh P2P Networking

MeshNetworkManager initializes Google Nearby Connections client with Strategy.P2P_CLUSTER. The startMesh() call begins advertising this device as an SCMS node (discoverable) AND scanning for other SCMS nodes simultaneously. When two devices connect, the offline device broadcasts queued complaints as Payload.fromBytes() JSON packets to the peer. If the receiving peer has internet, it calls OfflineComplaintManager.uploadRelayedComplaint() to upload the relayed complaint to the server.

Conclusion & Future Work

The Smart City Management System (SCMS) successfully delivers a production-quality, full-stack civic engagement platform that directly addresses the shortcomings of traditional infrastructure complaint systems. The project demonstrates how thoughtful integration of modern mobile technologies — Firebase, Jetpack Compose, Retrofit, CameraX, FCM, and Nearby Connections — can produce a system that is simultaneously powerful, offline-resilient, and citizen-friendly.

Key Achievements

  • ✅ Persistent authentication — zero login friction for returning users
  • ✅ GPS + photo verified complaints — fake or vague reports are structurally impossible
  • ✅ Offline-first queuing — citizenship is not gated by internet quality
  • ✅ Community-led prioritization via upvoting and duplicate detection
  • ✅ Real-time FCM alerts — authorities communicate instantly with all citizens
  • ✅ Gamified participation — leaderboard creates civic competitive culture
  • ✅ Admin web dashboard — complete complaint lifecycle management
  • ✅ P2P mesh relay — complaints survive even total internet infrastructure failure

Future Enhancements

  • 🤖 AI-based auto-categorization — automatically classify complaint type from submitted photo using ML Kit Vision
  • 📈 Predictive failure detection — ML model trained on historical complaints to predict next failure hotspots
  • 🌍 Multi-language support — internationalization for diverse urban populations
  • 🏛️ ERP integration — connect to municipal work order systems for automatic repair team dispatch
  • 📊 Analytics dashboard — heat maps, complaint volume trends, average resolution time KPIs
  • 🔄 Manual offline sync button — let users explicitly trigger sync from Profile screen
🎯

SCMS proves that a small team can build a complete, production-ready Smart City system using freely available cloud services, modern Android frameworks, and open-source tools — making the Smart City vision accessible and actionable even for emerging municipalities.
